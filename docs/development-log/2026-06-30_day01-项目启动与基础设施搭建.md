## 📌 跨平台命令说明(必读)

本文档涉及大量命令行操作。**Linux/macOS 和 Windows 的 shell 语法不同**——尤其是创建多行文本文件的命令。本文档采用以下约定:

### Shell 选择建议

| 操作系统 | 推荐 Shell | 原因 |
|---|---|---|
| **Linux** | bash / zsh | 系统默认 |
| **macOS** | zsh / bash | 系统默认 |
| **Windows** | **PowerShell**(推荐)| 比 CMD 强大,语法更现代 |
| Windows | Git Bash(替代) | 如果你装了 Git for Windows,可以直接用 bash 语法,跟 Linux 命令一致 |

> 💡 **Windows 用户的最佳选择**:
>
> **方案 A(推荐)**: 用 **PowerShell**——下文每个命令都给了 PowerShell 版本
>
> **方案 B(省事)**: 装 **Git for Windows** 后用其自带的 **Git Bash**——这样所有 Linux 命令都能直接跑,**完全不用看下面的 PowerShell 版本**
>
> **方案 C(不推荐)**: CMD——能力太弱,处理多行文本极其痛苦,本文不专门支持

### 命令格式约定

下文所有需要区分的命令,会用**并列代码块**形式给出:

````
🐧 Linux / macOS / Git Bash:
    <bash 命令>

🪟 Windows PowerShell:
    <PowerShell 命令>
````

**完全相同的命令**(如 `git`、`python`、`pip`)只写一次,因为它们跨平台一致。

### 路径分隔符

- Linux/macOS: `/`(正斜杠)
- Windows: `\`(反斜杠)或 `/`(PowerShell 也接受正斜杠)
- **Python 代码里**: 一律用 `pathlib.Path`,自动跨平台,无需关心

```

                ↓
      撞墙③: 哪些资产共享? 哪些私有? → APP_DIR 概念诞生
                ↓
      撞墙④: print 太弱 → 推出 logging_utils.py
                ↓
      撞墙⑤: 中英文对齐错位 → 推出 string_utils.py
                ↓
      锦上添花①: 想要环境快照 → 推出 system_utils.py
                ↓
      锦上添花②: 想要测耗时 → 推出 performance_utils.py
```

# 阶段 0: 项目基线(立规矩,先不建空目录)

D0 决定了我们用 Monorepo + apps/。今天第一件事:**立项目基线**——但**不预先建那 19 个目录**。

仔细想一下——**这 19 个目录,真的需要"上来一次性建好"吗?**

不需要。它们其实分 **3 类**,各自有自己的"出生时机":

```
┌─────────────────────────────────────────────────────────────┐
│  类别 A: 代码 / 文档目录                                     │
│    例如  apps/platform/src/od_platform/common/              │
│  出生时机: 写第一个文件时,编辑器自动建出沿途目录            │
│  进 git 吗: ✅ 进(随代码 commit)                            │
├─────────────────────────────────────────────────────────────┤
│  类别 B: 占位目录(为未来端预留)                            │
│    例如  apps/web-backend/                                   │
│  出生时机: 写它的 README 时顺带 mkdir                        │
│  进 git 吗: ✅ 进(README 进 git)                            │
├─────────────────────────────────────────────────────────────┤
│  类别 C: 运行时目录                                          │
│    例如  data/raw/, models/trained/, runs/                   │
│  出生时机: 由 init_project.py 在每个新环境创建               │
│  进 git 吗: ❌ 不进(内容 gitignored,clone 时不存在)        │
└─────────────────────────────────────────────────────────────┘
```

> 💡 **金句**: **"目录的【创建机制】= 它的【本质】"**

阶段 0 只做一件事:**立规矩**——把仓库根的几个"基线文件"放好。

## ▶ Step 0.2: 创建 marker file (.odp-workspace)

> 🤔 **思考题**: 为什么 marker file 要在仓库根**最早**就创建?

**答**: 因为 Stage 1 Monorepo 的目录深度让 `parent.parent.parent` 这种路径定位**几乎不可能用**(后面阶段 2 你会亲眼看到)。我们一开始就要用 marker file 模式——所以**先把 marker 文件放在仓库根**。

为什么 marker 文件名是 `.odp-workspace`?

- 名字**全球独一无二**,不会和别的项目冲突
- 用 `.` 开头意味着这是一个"配置 / 标记文件"(类似 `.git`、`.editorconfig`)
- "workspace" 这个词强调:这是 **workspace 根**(Monorepo 的根),不是某个 app 的根

**创建 marker file**:

🐧 **Linux / macOS / Git Bash**:
```bash
cat > .odp-workspace << 'EOF'
# ODPlatform Workspace Marker
#
# This empty file marks the workspace root.
# paths.py walks up from any module location to find this file,
# making path resolution robust against directory restructuring.
#
# Industry-standard pattern (used by pytest, poetry, git, npm, cargo).
#
# DO NOT DELETE THIS FILE.
EOF
```

🪟 **Windows PowerShell**:

```
@'
# ODPlatform Workspace Marker
#
# This empty file marks the workspace root.
# paths.py walks up from any module location to find this file,
# making path resolution robust against directory restructuring.
#
# Industry-standard pattern (used by pytest, poetry, git, npm, cargo).
#
# DO NOT DELETE THIS FILE.
'@ | Set-Content -Path .odp-workspace -Encoding UTF8
```

想用中文注释 + 兼容 PowerShell 5.1(Win10 自带版)也可以——下面这版用 .NET API 显式写无 BOM 的 UTF-8,**PS 5.1 / PS 7+ 都能跑**:

```powershell
$markerContent = @'
# ODPlatform 工作区标记文件
#
# 此空文件用于标记工作区根目录。
# paths.py 会从模块位置向上查找此文件，
# 使路径解析不依赖目录结构变化。
#
# 行业标准模式（pytest、poetry、git、npm、cargo 等均采用）。
#
# 请勿删除此文件。
'@

# 用 .NET API 显式写无 BOM 的 UTF-8(兼容 PS 5.1 / PS 7+)
[System.IO.File]::WriteAllText(
    "$PWD\.odp-workspace",
    $markerContent,
    (New-Object System.Text.UTF8Encoding $false)
)
```

> ⚠️ **不要直接写 `Set-Content -Encoding UTF8NoBOM`**——`UTF8NoBOM` 选项**只在 PowerShell 6+ 才有**,Win10 自带的 PS 5.1 上会报"不能将参数绑定到 -Encoding"。这就是上面用 .NET API 的原因。

> 📌 **PowerShell heredoc 语法说明**:
>
> - `@'...'@` 是 PowerShell 的"逐字字符串"(verbatim here-string),里面的内容**不做变量展开**——等价于 bash 的 `<< 'EOF'`(单引号包围的 EOF)
> - `@"..."@`(双引号版本)会展开 `$variable`——等价于 bash 的 `<< EOF`(无引号 EOF)
> - **本文档凡是 bash 用 `<< 'EOF'` 的地方,PowerShell 都用 `@'...'@`**
>
> **Encoding UTF8** 很重要——PowerShell 默认编码可能是 UTF-16 或带 BOM 的 UTF-8,会导致文件被 git 识别为二进制。我们显式指定 UTF-8。
>
> ⚠️ **PowerShell 5.1(Windows 10 自带版)的坑**: 它的 `-Encoding UTF8` 实际是"UTF-8 with BOM"。如果遇到乱码或 git diff 异常,可以用 `-Encoding utf8NoBOM`(需 PS 6+),或者改用下面的 .NET 方法:
> ```powershell
> [System.IO.File]::WriteAllText(
>     "$PWD\.odp-workspace",
>     @'
> # ODPlatform Workspace Marker
> # ...内容...
> '@,
>     (New-Object System.Text.UTF8Encoding $false)
> )
> ```

> 📌 **marker file 的"工业标准"地位**:
>
> 这种"放一个标志文件,从子目录向上找"的模式,主流工具都在用:
> - `pytest` 找 `pytest.ini` / `pyproject.toml`
> - `poetry` 找 `pyproject.toml`
> - `git` 找 `.git/`
> - `npm` 找 `package.json`
> - `cargo` 找 `Cargo.toml`

# 阶段 2: paths.py 的诞生(撞墙②③)

这是今天**最关键的一节**——paths.py 在 Stage 1 结构下,会**连续撞两面墙**:

- **撞墙②**: `parent.parent.parent...` 路径定位是死路 → marker file 模式
- **撞墙③**: 哪些资产共享给所有端? 哪些是端私有? → APP_DIR 概念

## ▶ Step 2.1: 设计 paths.py 的核心思想

paths.py 要回答 3 个问题:

| 问题 | 答案 |
|---|---|
| **ROOT_DIR 在哪?** | workspace 根。所有路径相对它 |
| **ROOT_DIR 怎么定位?** | (这里就是要撞墙的地方!) |
| **怎么暴露给上层?** | 大写常量(Python 约定) |

**关键决定**: **paths.py 自己不做 I/O,只定义路径常量** —— 这一刀分清楚,paths.py 就成为一个**纯定义模块**。

## ▶ Step 2.2: 第一版尝试 —— 数 parent(撞墙②)

按"最直接"的思路,我们会写:

```python
# apps/platform/src/od_platform/common/paths.py (v0 - 撞墙演示!)
from pathlib import Path

# 当前文件: apps/platform/src/od_platform/common/paths.py
# parent → common/
# parent.parent → od_platform/
# parent.parent.parent → src/
# parent.parent.parent.parent → platform/
# parent.parent.parent.parent.parent → apps/
# parent.parent.parent.parent.parent.parent → ODPlatform/  ← 终于到了!
ROOT_DIR: Path = Path(__file__).resolve().parent.parent.parent.parent.parent.parent
```

写到这一行,**你的反应应该是**:

> 🤔 "等等......这要数 6 个 parent?"
> 😱 "我数对了吗?"
> 🤬 "未来如果重构,common/ 移一层,我又要重数一遍?!"

**这,就是撞墙②**。

## ⚠ 撞墙②: parent.parent... 是死路

| 重构场景 | 需要的 parent 数 | 出错风险 |
|---|---|---|
| 现在 | 6 个 | 数错就 ROOT_DIR 错位 |
| 把 common/ 提到 od_platform 同层 | 5 个 | 改 1 处,容易漏 |
| 把 paths.py 移到 src/ 直接下面 | 4 个 | 改 1 处 |
| 加一层 namespace package | 7 个 | 改 1 处 |

**每一次目录调整都要数一遍 parent —— 这不是工程,是数学竞赛**。

## 🔧 修复: marker file 模式

回想阶段 0 我们已经创建了 `.odp-workspace` 这个 marker file ——**它就是为这一刻准备的**。

设计思想:

```
不数 parent。
改成"沿目录树往上爬,找一个标志文件"。
找到了,所在目录就是 ROOT_DIR。

无论文件位置怎么变,只要 marker 还在仓库根,定位永远正确。
```

# 阶段 3: APP_DIR 概念诞生(撞墙③ —— 共享 vs 私有 边界)

## ▶ Step 3.1: 一个看似简单的问题

> 🤔 **思考题**: 想象 paths.py 这样写:
>
> ```python
> LOGGING_DIR: Path = ROOT_DIR / "logging"
> ```
>
> 也就是说,**所有日志会写到** `ODPlatform/logging/` 下。
>
> 现在假设未来 V1.1 加了 web-backend 这个端——backend 跑 HTTP 服务,要写日志。
>
> **问题**: backend 的日志,应该往哪里写?

正确答案: **platform 写 platform 自己的目录,backend 写 backend 自己的目录**。

## ⚠ 撞墙③: 把"端私有"当成了"共享"

**"日志"的本质是【某个端运行时产生的副产品】——它和这个端绑定,不和别的端共享**。

类似的还有:

| 资源 | 共享还是私有? | 应该在哪 |
|---|---|---|
| `data/` | **共享**(所有端都用同一份数据集) | `ROOT_DIR/data/` |
| `models/` | **共享**(训练产出 → 推理服务/桌面端复用) | `ROOT_DIR/models/` |
| `runs/` | **共享**(各端都要查看训练结果) | `ROOT_DIR/runs/` |
| `logging/` | **私有**(端自己的运行时日志) | `APP_DIR/logging/` |
| `configs/` | **私有**(端自己的配置模板) | `APP_DIR/configs/` |
| `tests/` (端测试) | **私有** | `APP_DIR/tests/` |

→ **paths.py 必须区分这两类**。

> 💡 **金句**: **"目录的位置不是随便放的——放在哪一层,等于声明【这东西被谁共享】。"**

## 🔧 修复: 引入 APP_DIR 概念

paths.py 里加一个新概念:

```python
# 整个 workspace
ROOT_DIR = _find_workspace_root(...)

# 这一个端的根
APP_DIR = ROOT_DIR / "apps" / "platform"

# 共享(在 ROOT)
DATA_DIR  = ROOT_DIR / "data"
MODELS_DIR = ROOT_DIR / "models"

# 端私有(在 APP_DIR)
LOGGING_DIR = APP_DIR / "logging"
CONFIGS_DIR = APP_DIR / "configs"
```

## ▶ Step 3.2: paths.py 完整版(marker file + APP_DIR 一次到位)

> 💡 **创建文件**:
>
> 用编辑器在 `apps/platform/src/od_platform/common/paths.py` 创建文件——编辑器会自动建出 `common/` 目录。
>
> 如果你坚持用命令行先建目录:
>
> 🐧 Linux / macOS / Git Bash: `mkdir -p apps/platform/src/od_platform/common`
>
> 🪟 PowerShell: `New-Item -ItemType Directory -Path apps/platform/src/od_platform/common -Force`

写 `apps/platform/src/od_platform/common/paths.py`:

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : paths.py
# @Author    : 雨霓同学
# @Project   : ODPlatform
# @Function  : 集中定义项目路径常量
#              - 使用 marker file 模式定位 ROOT_DIR(workspace 根)
#              - 引入 APP_DIR 概念,把【共享资产】和【端私有资产】分开

from pathlib import Path
from typing import List, Tuple

# ============================================================
# Workspace 根目录定位 (marker file 模式)
# ============================================================
WORKSPACE_MARKER: str = ".odp-workspace"


def _find_workspace_root(
    start: Path,
    markers: Tuple[str, ...] = (WORKSPACE_MARKER,),
) -> Path:
    """
    从 start 开始沿父目录向上查找,返回包含任一 marker 文件的目录。

    Args:
        start: 起始路径(通常是 Path(__file__))
        markers: 一组 marker 文件名,任一存在即视为找到

    Returns:
        Path: workspace 根目录

    Raises:
        FileNotFoundError: 一直爬到文件系统根仍没找到
    """
    current = start.resolve()
    if current.is_file():
        current = current.parent

    for parent in [current, *current.parents]:
        for marker in markers:
            if (parent / marker).exists():
                return parent

    raise FileNotFoundError(
        f"找不到 workspace marker 文件 ({markers})。"
        f"请确认仓库根存在 {WORKSPACE_MARKER} 文件。"
    )


# 计算 ROOT_DIR(模块加载时执行一次)
ROOT_DIR: Path = _find_workspace_root(Path(__file__))


# ============================================================
# 端根目录 APP_DIR (platform 这一个端的根)
# ============================================================
APP_DIR: Path = ROOT_DIR / "apps" / "platform"


# ============================================================
# 【共享资产】(在 ROOT_DIR 下,所有端可访问)
# ============================================================
DATA_DIR: Path = ROOT_DIR / "data"
MODELS_DIR: Path = ROOT_DIR / "models"
RUNS_DIR: Path = ROOT_DIR / "runs"

# 模型子目录
PRETRAINED_MODELS_DIR: Path = MODELS_DIR / "pretrained"  # 下载的预训练权重(输入)
TRAINED_MODELS_DIR: Path = MODELS_DIR / "trained"        # 训练产出、长期保留的权重(产物)

# 数据集子目录
# 设计依据见 D0 决定 5:原始(只读)与派生(可再生)分家。
# train/val/test 划分属于派生产物,会在数据处理那天用固定种子一次性
# 冻结进 processed/ 内;init 阶段只建 raw / processed 两个空目录。
RAW_DATA_DIR: Path = DATA_DIR / "raw"              # 原始数据:只读,绝不就地改
PROCESSED_DATA_DIR: Path = DATA_DIR / "processed"  # 派生数据集:含冻结后的 train/val/test


# ============================================================
# 【端私有资产】(在 APP_DIR 下,只属于 platform 这个端)
# ============================================================
CONFIGS_DIR: Path = APP_DIR / "configs"
LOGGING_DIR: Path = APP_DIR / "logging"
UNIT_TEST_DIR: Path = APP_DIR / "tests"


# ============================================================
# 【顶层文档目录】(共享给所有人)
# ============================================================
DOCS_DIR: Path = ROOT_DIR / "docs"


# ============================================================
# 【工程基础设施目录】(共享)
# ============================================================
SCRIPTS_DIR: Path = ROOT_DIR / "scripts"


# ============================================================
# 对外暴露的"要初始化的目录列表"
# ============================================================
def get_dirs_to_initialize() -> List[Path]:
    """
    返回项目启动时需要确保存在的所有目录列表。

    这是 init_project.py 的【唯一数据源】——
    paths.py 决定要哪些目录,init_project 只负责创建。

    Returns:
        所有需要初始化的目录路径列表
    """
    return [
        # 共享资产
        DATA_DIR,
        RUNS_DIR,
        MODELS_DIR,
        PRETRAINED_MODELS_DIR,
        TRAINED_MODELS_DIR,
        RAW_DATA_DIR,
        PROCESSED_DATA_DIR,
        # 端私有资产
        CONFIGS_DIR,
        LOGGING_DIR,
        UNIT_TEST_DIR,
        # 工程基础设施
        SCRIPTS_DIR,
        DOCS_DIR,
    ]


if __name__ == "__main__":
    print(f"ROOT_DIR (workspace) = {ROOT_DIR}")
    print(f"APP_DIR  (platform)  = {APP_DIR}")
    print(f"\n共享资产:")
    print(f"  DATA_DIR    = {DATA_DIR.relative_to(ROOT_DIR)}")
    print(f"  MODELS_DIR  = {MODELS_DIR.relative_to(ROOT_DIR)}")
    print(f"  RUNS_DIR    = {RUNS_DIR.relative_to(ROOT_DIR)}")
    print(f"\n端私有资产:")
    print(f"  CONFIGS_DIR = {CONFIGS_DIR.relative_to(ROOT_DIR)}")
    print(f"  LOGGING_DIR = {LOGGING_DIR.relative_to(ROOT_DIR)}")

    print(f"\n要初始化的目录共 {len(get_dirs_to_initialize())} 个:")
    for d in get_dirs_to_initialize():
        print(f"  - {d.relative_to(ROOT_DIR)}")
```

> 📌 **这份最终清单,和阶段 0 那句"只运行时目录需要 init"对得上吗?——值得停一秒。**
>
> 阶段 0 我们立过一条**概念**上的三分法:运行时目录得靠 init 创建,代码 / 文档那类"随代码进 git、自然就在"。没错。但你看这份**最终**清单,它已经按 D0 的"共享 / 私有 / 基础设施"长全了——里头既有运行时产物,也有**珍贵输入**(`raw`、`pretrained`),还有**进 git 的**代码 / 文档 / 配置(`configs`、`tests`、`scripts`、`docs`)。
>
> 为什么把"随代码出生"的那些也列进来?因为对 `init` 来说,它干的事只有一件——**确保该在的目录都在**。`mkdir(exist_ok=True)` 对"已经在的"是无害的 no-op,对"该在却没在的"(比如一个刚 clone、配置目录还空着的环境)补上。所以一视同仁地列全,既让半残的工作区能一键补齐,也让后面的工具(pytest 找 `tests/`、数据流水线往 `configs/datasets/` 写)都有地方落脚。
>
> **但请记住这个"混放"。** 这份清单把"删了能再生的产物"和"删了就完蛋的输入 / 代码"焊在了一起。对 `init`(只增不减)无所谓;可一旦有人想"反过来用"它做项目重置——**"能不能盲目反向删这份清单"会成为下一篇 D2 的第一堵墙**。同一份清单,正向 `mkdir` 无害、反向 `rmtree` 致命,这正是"危险操作不对称"的活例子。

>

> 如果想要"用完即恢复"的效果,PowerShell 里可以这样:
> ```powershell
> $oldPath = $env:PYTHONPATH

## ▶ Step 5.1: 设计目标

我们要做一个**比 print 强 10 倍**的日志工具。但在写代码之前,先把**整套 logger 设计哲学**讲透——这是 D1 最重要的一节,后续所有模块都依赖它。

### 5.1.1 业务模块怎么"发声"?

Python 的 logging 模块有一个**标准做法**:每个业务模块顶部写一行——

```python
import logging
logger = logging.getLogger(__name__)
```

`__name__` 是 Python 内建变量,在每个模块里会**自动展开成该模块的点号全路径**。例如:

| 文件位置 | `__name__` 的值 |
|---|---|
| `apps/platform/src/od_platform/cli/init_project.py` | `"od_platform.cli.init_project"` |
| `apps/platform/src/od_platform/common/paths.py` | `"od_platform.common.paths"` |
| `apps/platform/src/od_platform/common/system_utils.py` | `"od_platform.common.system_utils"` |

这些 logger 名构成一棵**以 `od_platform` 为根的树**。

### 5.1.2 谁来"配 handler"?

Python logger 的**冒泡机制**: 一条日志从子 logger 一路冒泡到祖先 logger,**任何一层挂了 handler, 就会被该 handler 处理**。

我们的设计:
- **业务模块**(paths / system_utils / init_project / 未来的 train / val ...): 只写 `logger = logging.getLogger(__name__)`——**不配 handler, 也不知道 handler 长什么样**, 只管 `logger.info("...")` 发声
- **基础设施模块**(`logging_utils.py`): 提供一个 `get_logger()` 函数, 把 console + file 两个 handler 挂到 **根 logger `"od_platform"`** 上
- **CLI 入口**(`init_project.py` 的 `initialize_project()` 函数最开头): **调用一次** `get_logger()` 把 handler 装好——之后整个进程里所有 `getLogger(__name__)` 拿到的子 logger, 都会通过冒泡机制走到根 logger, 由根 logger 的 handler 统一处理

> 💡 **金句**: **"业务模块要做的事是【发声】, 不是【配置音响】。把'怎么配 handler'留给基础设施, 业务模块只关心'我要说什么'。"**

### 5.1.3 这样设计能拿到什么?

1. **零配置**——业务模块顶部一行 `logger = getLogger(__name__)`, 不用关心 handler 怎么配
2. **可按子树调级别**——未来某天想 debug converter 但不想刷屏 splitter:
   ```python
   logging.getLogger("od_platform.data_converter").setLevel(logging.DEBUG)
   logging.getLogger("od_platform.data_splitter").setLevel(logging.WARNING)
   ```
3. **Python 圈惯用法**——任何成熟 Python 项目读起来都是这个味道, 新人零障碍
4. **日志里能看出来源**——`getLogger(__name__)` 的 logger 名会自动出现在日志格式 `%(name)s` 字段里, 一眼能看到"这条日志来自哪个模块"

我们今天就把这套设计**一次到位**——后续每一篇写任何业务模块, 顶部都是 `logger = logging.getLogger(__name__)`, 永不变化。

## ▶ Step 5.3: 写 logging_utils.py(完整代码)

**用编辑器**创建 `apps/platform/src/od_platform/common/logging_utils.py`:

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : logging_utils.py
# @Author    : 雨霓同学
# @Project   : ODPlatform
# @Function  : 高度可配置的日志工具,创建独立日志文件 + 彩色控制台输出
#
# 设计哲学:
#   - 业务模块: 顶部一行 logger = logging.getLogger(__name__), 不配 handler
#   - 本模块: 提供 get_logger(), 把 handler 挂到根 logger "od_platform" 上
#   - CLI 入口: 调用一次 get_logger() 完成 handler 装配,
#               之后所有 getLogger(__name__) 通过冒泡机制自动继承

import logging
import sys
import platform
from datetime import datetime
from pathlib import Path
from typing import Optional
from colorlog import ColoredFormatter


# 项目根 logger 名 = 顶层 Python 包名
# 业务模块写 getLogger("od_platform.xxx.yyy") 会通过冒泡找到这里挂的 handler
ROOT_LOGGER_NAME: str = "od_platform"


def get_logger(
    base_path: Path,
    log_type: str = "general",
    model_name: Optional[str] = None,
    log_level: int = logging.INFO,
    temp_log: bool = False,
    encoding: str = "utf-8",
    logger_name: str = ROOT_LOGGER_NAME,
) -> logging.Logger:
    """
    配置项目【根 logger】, 挂上 console + file handler。

    设计要点:
        - 默认 logger_name = "od_platform" (顶层包名),
          与业务模块的 getLogger(__name__) 形成 logger 树
        - 业务模块的日志会通过冒泡机制自动来到这里, 被本函数挂的 handler 处理
        - 重复调用幂等 (检查 handlers 是否已配置, 避免重复挂)

    Args:
        base_path: 日志根目录(通常是 paths.LOGGING_DIR)
        log_type: 日志类型,如 "init_project" / "train" / "val"
        model_name: 模型名(如 "yolo11n"),会拼到日志文件名里
        log_level: 日志级别(默认 INFO)
        temp_log: 是否标记为临时日志(文件名前缀变成 "temp")
        encoding: 文件编码(默认 utf-8)
        logger_name: 要配置的 logger 名 (默认是根 logger "od_platform";
                     特殊场景可以传别的名字配出独立 logger, 比如某个子系统想要
                     "od_platform.audit.xxx" 这样的隔离子树, 让它的日志单独成一棵、
                     不与主 logger 混在一起)

    Returns:
        配置好的 logging.Logger 实例
    """
    # ============================================================
    # 1. 获取/复用命名 Logger
    # ============================================================
    logger = logging.getLogger(logger_name)
    # 幂等保护: getLogger 是 singleton——同一个 logger_name 多次调用拿到同一对象。
    # 第一次调用配置好 handler 后, 后续调用直接返回, 避免重复挂多份 handler
    # (重复挂 handler 会导致同一条日志被打印 N 遍)。
    #
    # 副作用: 第二次调用即使传了不同的 log_level / temp_log 也不再生效——这是有意为之。
    # 整个进程只配置一次日志, 由 CLI 入口在最早时机调用一次完成。
    if logger.handlers:
        return logger

    logger.setLevel(log_level)
    logger.propagate = False  # 根 logger 不再向上冒泡(避免与 Python 默认 root logger 重复输出)

    # ============================================================
    # 2. 准备日志目录
    # ============================================================
    log_dir: Path = base_path / log_type
    log_dir.mkdir(parents=True, exist_ok=True)

    # ============================================================
    # 3. 构造日志文件名: <prefix>_<timestamp>[_<model>].log
    # ============================================================
    timestamp: str = datetime.now().strftime("%Y%m%d-%H%M%S-%f")[:21]
    prefix = "temp" if temp_log else log_type.replace("_", "-")

    filename_parts = [prefix, timestamp]
    if model_name:
        safe_model = "".join(
            c if c.isalnum() or c in "_-" else "_" for c in model_name
        )
        filename_parts.append(safe_model)
    log_file: Path = log_dir / ("_".join(filename_parts) + ".log")

    # ============================================================
    # 4. 文件 Handler(完整格式, 含 logger 名)
    # ============================================================
    # %(name)s 字段会输出 logger 名, 比如 "od_platform.cli.init_project"
    # 这是 getLogger(__name__) 设计带来的福利——一眼看出日志来源模块
    file_formatter = logging.Formatter(
        fmt="%(asctime)s - %(name)s - %(levelname)-8s - "
            "%(filename)s:%(lineno)d - %(funcName)s - %(message)s",
        datefmt="%Y-%m-%d %H:%M:%S",
    )
    file_handler = logging.FileHandler(log_file, encoding=encoding)
    file_handler.setLevel(log_level)
    file_handler.setFormatter(file_formatter)
    logger.addHandler(file_handler)

    # ============================================================
    # 5. 控制台 Handler(彩色 + 紧凑格式)
    # ============================================================
    console_formatter = ColoredFormatter(
        "%(log_color)s%(asctime)s%(reset)s "
        "%(log_color)s[%(levelname)-8s]%(reset)s "
        "%(cyan)s%(filename)-25s%(reset)s:"
        "%(blue)s%(lineno)-4d%(reset)s "
        "%(log_color)s│ %(message)s%(reset)s",
        datefmt="%Y-%m-%d %H:%M:%S",
        log_colors={
            "DEBUG":    "white",
            "INFO":     "green",
            "WARNING":  "yellow",
            "ERROR":    "red",
            "CRITICAL": "bold_red,bg_white",
        },
        style='%'
    )
    console_handler = logging.StreamHandler(sys.stdout)
    console_handler.setLevel(log_level)
    console_handler.setFormatter(console_formatter)
    logger.addHandler(console_handler)

    # ============================================================
    # 6. 初始化信息
    # ============================================================
    logger.info('=' * 60)
    logger.info(f"日志系统初始化完成")
    logger.info(f"运行环境: {platform.system()} {platform.release()}")
    logger.info(f"阶段类型: {log_type}")
    logger.info(f"日志文件: {log_file}")
    logger.info(f"日志级别: {logging.getLevelName(log_level)}")
    logger.info(f"模型名称: {model_name or '无'}")
    logger.info('=' * 60)

    return logger


if __name__ == "__main__":
    # 模块自测——演示"基础设施装 handler + 业务模块发声"的标准用法
    from od_platform.common.paths import LOGGING_DIR

    # 第一步: 装配根 logger 的 handler(这是 CLI 入口该做的事, 这里只是自测演示)
    get_logger(
        base_path=LOGGING_DIR,
        log_type="test",
        temp_log=True,
    )

    # 第二步: 业务代码就这样写——拿一个 __name__ logger, 直接发声
    # 注意: 这里 __name__ 是 "__main__"(因为直接跑这个文件),
```

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : string_utils.py
# @Author    : 雨霓同学
# @Project   : ODPlatform
# @Function  : 字符串工具——CJK 感知的对齐 / 表格格式化


def get_display_width(text: str) -> int:
    """
    计算字符串的实际显示宽度。

    Examples:
        >>> get_display_width("hello")
        5
        >>> get_display_width("你好")
        4
        >>> get_display_width("hi你好")
        6
    """
    return sum(2 if _is_wide_char(c) else 1 for c in text)


def _is_wide_char(char: str) -> bool:
    """判断单字符是否为宽字符(占 2 个显示宽度)"""
    if '\u4e00' <= char <= '\u9fff':  # 中日韩统一表意文字
        return True
    if '\u3000' <= char <= '\u303f':  # 中日韩标点符号
        return True
    if '\uff00' <= char <= '\uffef':  # 全角 ASCII
        return True
    if '\u3400' <= char <= '\u4dbf':  # 中日韩扩展 A
        return True
    if '\u3040' <= char <= '\u309f':  # 平假名
        return True
    if '\u30a0' <= char <= '\u30ff':  # 片假名
        return True
    if '\uac00' <= char <= '\ud7af':  # 韩文音节
        return True
    return False


def pad_to_width(text: str, width: int, align: str = 'left') -> str:
    """
    将字符串填充到指定显示宽度。

    Args:
        text: 输入字符串
        width: 目标显示宽度
        align: 'left' / 'right' / 'center'

    Examples:
        >>> pad_to_width("hello", 10)
        'hello     '
        >>> pad_to_width("你好", 10)
        '你好      '
    """
    current = get_display_width(text)
    padding = width - current
    if padding <= 0:
        return text

    if align == 'right':
        return ' ' * padding + text
    elif align == 'center':
        left = padding // 2
        right = padding - left
        return ' ' * left + text + ' ' * right
    else:
        return text + ' ' * padding


def format_table_row(columns: list, widths: list, aligns: list = None) -> str:
    """格式化表格一行"""
    if aligns is None:
        aligns = ['left'] * len(columns)
    assert len(columns) == len(widths) == len(aligns), "列数 / 宽度 / 对齐数必须一致"

    parts = [
        pad_to_width(str(col), w, a)
        for col, w, a in zip(columns, widths, aligns)
    ]
    return ' '.join(parts)


def format_table_separator(widths: list, char: str = '-') -> str:
    """生成与表格列宽匹配的分隔线"""
    total = sum(widths) + len(widths) - 1
    return char * total


if __name__ == "__main__":
    print("=== string_utils 自测 ===\n")

    cases = [("hello", 5), ("你好", 4), ("hi你好", 6), ("类别名称", 8)]
    for text, expected in cases:
        actual = get_display_width(text)
        status = "✓" if actual == expected else "✗"
        print(f"  {status} '{text}' → {actual} (期望 {expected})")

    print("\n=== 表格对齐示例 ===")
    widths = [12, 20, 6]
    aligns = ['left', 'left', 'right']
    print(format_table_row(['ID', '类别名称', '数量'], widths, aligns))
    print(format_table_separator(widths))
    print(format_table_row(['0', 'head', '60'], widths, aligns))
    print(format_table_row(['1', 'ordinary_clothes', '59'], widths, aligns))
    print(format_table_row(['2', '反光衣', '116'], widths, aligns))
```

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : system_utils.py
# @Author    : 雨霓同学
# @Project   : ODPlatform
# @Function  : 环境快照——OS / Python / PyTorch / CPU / GPU / 内存

from __future__ import annotations

import os
import platform
import time
import logging
from typing import Optional

from od_platform.common.string_utils import pad_to_width

# 业务模块标准写法
logger = logging.getLogger(__name__)


def _format_size(bytes_size) -> str:
    """简洁的字节单位格式化"""
    if not bytes_size or not isinstance(bytes_size, (int, float)):
        return "N/A"
    if bytes_size >= 1024 ** 3:
        return f"{bytes_size / (1024 ** 3):.2f} GB"
    if bytes_size >= 1024 ** 2:
        return f"{bytes_size / (1024 ** 2):.2f} MB"
    return f"{bytes_size / 1024:.2f} KB"


def get_basic_device_info() -> dict:
    """返回结构化的环境信息字典"""
    cpu_name = platform.processor() or platform.machine() or "Unknown CPU"
    cpu_cores = os.cpu_count() or "Unknown"

    # 内存(psutil 是软依赖)
    try:
        import psutil
        memory = psutil.virtual_memory()
        total_ram = _format_size(memory.total)
        available_ram = _format_size(memory.available)
        ram_usage = f"{memory.percent:.1f}%"
    except ImportError:
        total_ram = "Unknown (psutil 未安装)"
        available_ram = "Unknown"
        ram_usage = "Unknown"

    # PyTorch / Ultralytics(软依赖)
    try:
        import torch
        torch_version = torch.__version__
        cuda_available = torch.cuda.is_available()
        gpu_count = torch.cuda.device_count() if cuda_available else 0
        _torch = torch  # 缓存模块引用, 后面 GPU 详情段直接用, 不再重复 import
    except ImportError:
        torch_version = "Unknown (torch 未安装)"
        cuda_available = False
        gpu_count = 0
        _torch = None

    try:
        from ultralytics import __version__ as ultralytics_version
    except ImportError:
        ultralytics_version = "Unknown (ultralytics 未安装)"

    gpu_info = {
        "CUDA可用": cuda_available,
        "GPU数量": gpu_count,
    }
    if cuda_available and _torch is not None:
        for i in range(gpu_count):
            gpu_info[f"GPU_{i}"] = _torch.cuda.get_device_name(i)
            gpu_info[f"GPU_{i}_显存"] = _format_size(
                _torch.cuda.get_device_properties(i).total_memory
            )

    return {
        "系统信息": {
            "操作系统": f"{platform.system()} {platform.release()} ({platform.machine()})",
            "主机名": platform.node(),
            "Python版本": platform.python_version(),
            "PyTorch版本": torch_version,
            "Ultralytics版本": ultralytics_version,
            "当前时间": time.strftime("%Y-%m-%d %H:%M:%S"),
        },
        "CPU信息": {
            "CPU型号": cpu_name,
            "核心数": cpu_cores,
        },
        "内存信息": {
            "总内存": total_ram,
            "可用内存": available_ram,
            "使用率": ram_usage,
        },
        "GPU信息": gpu_info,
    }


def log_device_info(target_logger: Optional[logging.Logger] = None) -> dict:
    """把环境信息打印到 logger。

    Args:
        target_logger: 可选, 指定的 logger 实例。
                       None 时使用本模块的 __name__ logger(推荐——
                       业务方调用时, 通过冒泡机制走到根 logger 的 handler)。
    """
    log = target_logger if target_logger is not None else logger

    info = get_basic_device_info()

    log.info("🚀 运行环境概览".center(60))
    log.info("=" * 60)

    key_width = 16

    for category, details in info.items():
        log.info(f"{category}".center(60))
        log.info("-" * 60)
        for key, value in details.items():
            padded_key = pad_to_width(key, key_width)
            log.info(f"  {padded_key}: {value}")

    return info


if __name__ == "__main__":
    # 自测——这里用 basicConfig 是因为自测脚本里没走完整 CLI 入口,
    # 真实使用中, CLI 入口已经调过 get_logger(), 这里不需要做任何配置
    logging.basicConfig(level=logging.INFO, format="%(message)s")
    log_device_info()
```

```python
#!/usr/bin/env python
# -*- coding:utf-8 -*-
# @FileName  : performance_utils.py
# @Author    : 雨霓同学
# @Project   : ODPlatform
# @Function  : 性能测量工具——@time_it 装饰器 (name 支持 callable 的兼容版)

import logging
import time
from functools import wraps
from typing import Callable, Optional, Union

# 业务模块标准写法
logger = logging.getLogger(__name__)


def time_it(
    iterations: int = 1,
    name: Optional[Union[str, Callable[..., str]]] = None,
    logger_instance: logging.Logger = None,
):
    """
    通用执行时间测量装饰器。

    name 可以传:
        - None      : 用被装饰函数的 __name__ (老行为, 不变)
        - str       : 固定显示名 (老行为, 不变)
        - callable  : 一个接收"被装饰函数运行期实参"的函数, 返回显示名 (新增)
                      用于每次调用名字不同的场景, 例如 D4 的逐 check 计时:
                          @time_it(name=lambda entry, ctx: entry.name)
                          def _safe_run_one(entry, ctx): ...

    Examples:
        @time_it()
        def my_func(): ...

        @time_it(iterations=10, name="批量推理")
        def infer_batch(): ...

        # 指定 logger:
        @time_it(logger_instance=logging.getLogger("my_module"))
        def custom(): ...

        # 运行期动态取名 (callable):
        @time_it(name=lambda entry, ctx: f"check:{entry.name}", logger_instance=logger)
        def _safe_run_one(entry, ctx): ...
    """
    # 默认用本模块的 __name__ logger;
    # 调用方可以传一个特定 logger 进来(比如希望日志带着调用方的 logger 名)
    log = logger_instance if logger_instance is not None else logger

    def _format_time_auto_unit(seconds: float) -> str:
        """自动选择合适单位"""
        if seconds < 0.001:
            return f"{seconds * 1_000_000:.3f} 微秒"
        elif seconds < 1.0:
            return f"{seconds * 1000:.3f} 毫秒"
        elif seconds < 60:
            return f"{seconds:.2f} 秒"
        elif seconds < 3600:
            mins = seconds // 60
            secs = seconds % 60
            return f"{mins:.0f} 分钟 {secs:.2f} 秒"
        else:
            hours = seconds // 3600
            mins = (seconds % 3600) // 60
            secs = (seconds % 3600) % 60
            return f"{hours:.0f} 小时 {mins:.0f} 分钟 {secs:.2f} 秒"

    def _resolve_name(func, args, kwargs) -> str:
        # 只有 callable 才走"运行期取名"; str / None 完全保持老逻辑
        if callable(name):
            try:
                return name(*args, **kwargs)
            except Exception:
                # 取名失败绝不能拖垮被测函数 —— 退回函数名, 测量照常进行
                log.warning(f"time_it: name() 计算失败, 回退到 {func.__name__}", exc_info=True)
                return func.__name__
        if name is not None:
            return name
        return func.__name__

    def decorator(func):
        @wraps(func)
        def wrapper(*args, **kwargs):
            display_name = _resolve_name(func, args, kwargs)
            total = 0.0
            result = None
            for _ in range(iterations):
                start = time.perf_counter()
                result = func(*args, **kwargs)
                end = time.perf_counter()
                total += (end - start)

            avg = total / iterations
            avg_str = _format_time_auto_unit(avg)

            if iterations == 1:
                log.info(
                    f"性能报告: '{display_name}' 执行 {iterations} 次 ,  耗时: {avg_str}"
                )
            else:
                total_str = _format_time_auto_unit(total)
                log.info(
                    f"性能报告: '{display_name}' 执行 {iterations} 次 | "
                    f"总耗时: {total_str} | 平均耗时: {avg_str}"
                )

            return result
        return wrapper
    return decorator


if __name__ == "__main__":
    # 自测时单独配一个简单 handler——真实使用中, CLI 入口已经配好了, 这里不用
    logging.basicConfig(level=logging.INFO, format="%(asctime)s - %(message)s",
                        datefmt="%H:%M:%S")

    @time_it(name="测试睡眠 0.1 秒")
    def test_sleep():
        time.sleep(0.1)

    @time_it(iterations=5, name="测试快函数")
    def test_fast():
        sum(range(1000))

    test_sleep()
    test_fast()
```

```python
def _check_raw_data_status() -> List[str]:
    """检查 raw 数据目录的状态"""
    raw_status: List[str] = []
    rel_raw = RAW_DATA_DIR.relative_to(ROOT_DIR)

            raw_status.append(f"  • {sub.name}")

                raise SystemExit(1) from e
        else:
            logger.info(f"目录已存在: {rel}")
            existed.append(d)

    logger.info("项目核心目录检查创建完毕".center(LINE_WIDTH, '='))

    # 3. 检查原始数据目录状态
    logger.info("开始检查原始数据目录".center(LINE_WIDTH, '='))

    # 4. 输出汇总信息

    for d in existed:

        ))

    if not created and not existed:
        logger.info("(本次没有任何目录变化)")

    logger.info("原始数据目录状态:")
    for status in raw_status:
        logger.info(f"  - {status}")

    logger.info("项目初始化完毕".center(LINE_WIDTH, '='))
    logger.info("下一步:把数据集放到 data/raw/ 下,然后运行数据转换脚本")
    logger.info("=" * LINE_WIDTH)


if __name__ == "__main__":
    initialize_project()
```

# 阶段 11: 工程化 —— pyproject.toml (hatchling) + entry-point script

到这里我们已经完成了**功能上的 init_project**。但它还有一个尴尬的地方:**必须用 `python scripts/init_project.py` 来跑,而不是一个干净的命令**。

更专业的做法是:把 platform 装成一个 **pip-installable package**,然后通过命令行直接调用:

```
odp-init    # 这是我们要做的目标命令
```

## ▶ Step 11.1: 关于 build backend 的选择

在写 `pyproject.toml` 之前, 30 秒讲一下 **build backend**。

Python 的现代打包标准是 **PEP 517**(2017 定稿)——它把"打包"从一个固定行为变成了可插拔的"backend":

```toml
# pyproject.toml
[build-system]
requires = ["<backend>"]        # ← 你声明用哪个 backend
build-backend = "<backend.module>"
```

主流选项:

| Backend | 特点 |
|---|---|
| `setuptools` | 老牌, 生态最广, 但有"egg-info 钻进源码树"的历史包袱 |
| **`hatchling`** | **现代, 干净, PEP 660 native (无 egg-info), 我们选这个** |
| `flit-core` | 极简, 纯 Python 包专用 |
| `poetry-core` | 配套 Poetry 工作流 |

**为什么选 hatchling**:

1. **源码树干净** —— `pip install -e .` 之后, `src/` 下**不会冒出一个 `od_platform.egg-info/` 目录**。元数据写到 `site-packages/` 的 `*.dist-info/` 里, 跟项目源码彻底分离
2. **现代标准** —— PEP 660 native editable install, 无历史包袱
3. **更严格** —— 配置写错会**早期报错**, 而 setuptools 经常默默吞掉小错装出一个空包

> 💡 **金句**: **"egg-info 不是 Python 的问题, 是 setuptools 的问题。"** 我们直接选不产生 egg-info 的 backend。

### 🔍 仔细看最下面那两段 `[tool.hatch.*]` 配置

整个 pyproject.toml 真正"换了 backend 才需要动"的, 只有最下面这两段。**`[project]` 和 `[project.scripts]` 一个字都没改**——因为它们是 **PEP 621 标准字段**, 跟 backend 无关。换到 flit-core / poetry-core 也照样能用。

| 你声明的事 | 给 hatchling 的写法 |
|---|---|
| 包源码在 `src/od_platform/` | `[tool.hatch.build.targets.wheel]`<br>`packages = ["src/od_platform"]` |
| 版本号动态从 `_version.py` 读 | `[tool.hatch.version]`<br>`path = "src/od_platform/_version.py"` |

就这么多——**两段配置**。相比 setuptools 同等行为需要的三段(`[tool.setuptools]` + `[tool.setuptools.packages.find]` + `[tool.setuptools.dynamic]`),hatchling 的表达力**更直接**:**直接指源码路径,而不是"配一堆规则让它去 find"**。

> 📌 **教学要点**: 这就是 PEP 517 设计的美妙之处——**"mechanism(backend 怎么干)和 policy(项目元数据)分离"**。
> 你描述项目用 `[project]`,backend 怎么干用 `[tool.<backend>.*]`。**换 backend 不动 `[project]`**——这条原则会在你未来读到的每一份现代 pyproject.toml 里都成立。

> 💡 **对比**: 如果当初选了 setuptools,这里会多出一个 `od_platform.egg-info/` 目录。我们提前避开了这个历史包袱。


🎯 **顿悟时刻**: 你刚在 `pyproject.toml` 里写的那行——

```toml
[project.scripts]
odp-init = "od_platform.cli.init_project:initialize_project"
```

**就被 hatchling 翻译成这一行 `entry_points.txt`**。pip 看到这个文件,生成了一个调度脚本(Linux 上在 `<env>/bin/odp-init`,Windows 上在 `<env>/Scripts/odp-init.exe`),那个脚本的**全部工作**就是:**找到 `od_platform.cli.init_project` 模块,调用它的 `initialize_project` 函数**。

**这就是 entry-point 的全部魔法**——没有黑盒,就一个文本配置文件。

> 💡 **回想 setuptools 时代**: 同样的 `entry_points.txt` 会被放到 `apps/platform/src/od_platform.egg-info/` 里(钻进源码树)。
> hatchling 把它挪到了 `site-packages/<pkg>-<ver>.dist-info/`(在 Python 环境里,**不在你的项目里**)。
> **文件内容完全一样,只是放对了地方**——这就是 PEP 660 解决的问题。

## ▶ Step 11.5: 三种调用方式的总结

```
# 方式 1: entry-point 命令(生产推荐)
odp-init

# 方式 2: 模块路径
python -m od_platform.cli.init_project

# 方式 3: 仓库根脚本
python scripts/init_project.py
```

| 方式 | 优点 | 缺点 | 何时用 |
|---|---|---|---|
| 1. odp-init | 最简洁 | 需要先 `pip install -e .` | **生产 / 团队共享** |
| 2. python -m | 不需要安装 | 命令长 | 临时测试 |
| 3. python scripts/ | 不需要配 PYTHONPATH | "看起来像脚本" | **新人首次跑 / CI** |

> 📌 **常见疑问**: "既然装完之后能 `python -m od_platform.cli.init_project`,为什么还要 entry-point?"
>
> 答:这两条路解决的是**不同问题**。
> - `python -m` 走的是 **Python 的模块解析**——只要包能 import,就能 `-m` 调用。**不依赖 entry_points.txt**。
> - entry_points 解决的是**让用户在 shell 里不写 `python -m`,直接敲一个短命令** `odp-init` 就跑起来。
>
> 两条路并存,互不冲突。生产环境给用户的永远是短命令。
